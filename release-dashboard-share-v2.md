## 一、背景：为什么要做这个

学练机迭代节奏快，版本多（目前已有 22 个版本、159 个技术功能项），发版信息分散在两个地方：

- **Confluence**：技术功能清单、RD/QA 负责人、测试状态（页面 ID：917724867）
- **石墨文档**：产品功能说明（GUID：AZlp79EvK4zQHR69）

每次想看「这个版本上了什么」，要同时打开两个平台，费时费力，也难以全局把握进度。

**目标**：做一个聚合 Dashboard，把两个数据源合并成一个可视化页面，随时查、随时刷新。

## 三、与 AI 协作的方式

### 3.1 总体协作模式

这个项目**全程没有手写代码**，通过和 AI（OpenClaw）对话完成。协作分工如下：

**我负责：**
- 决定要做什么、做成什么样
- 在浏览器看效果，发现哪里不对
- 提供数据源（Confluence 链接、石墨 GUID、工单 Cookie 等）
- 做最终的验收判断

**AI 负责：**
- 根据描述生成代码
- 解析数据源（Confluence API、石墨文档）
- 修 bug、调样式、加功能
- 服务器部署（SSH 连接服务器操作）

```
我描述需求 → AI 生成代码 → 我在浏览器看效果 → 发现问题 → 告诉 AI → AI 修复/迭代
```

### 3.2 各阶段协作过程

**阶段 1：从 Confluence 拉数据，生成第一版**

提供 Confluence 页面链接，告诉 AI 表格的列结构（功能、RD、QA、备注、Jira），AI 通过 API 抓取、解析，生成第一版静态 HTML（41KB，27个版本，100个功能项）。

*关键：只需描述「我想看什么」，不需要自己读 Confluence API 文档。*

---

**阶段 2：接入石墨文档（双数据源）**

石墨文档结构比 Confluence 复杂，存在大量噪音数据（展示建议、宣传文案、FAQ 等），AI 迭代了 5 个版本的解析器（v1-v5），最终干净解析出 11 个版本的产品功能。

关键解析决策：
- **detail_parts 结构**：每个功能拆成 `{label, text}` 列表
- **噪音合并策略**：无机型+无实质内容的条目合并到前一个功能，或过滤
- **重复版本去重**：同一石墨 block 只挂载到最早的 Confluence 版本

*关键：遇到解析不干净，把结果发给 AI 看，说「这条不对」，它自己找规律修。*

---

**阶段 3：加工单 tab（拉取外部系统数据）**

工单数据来自内部系统，需要登录态（Cookie）。我直接把 Cookie 发给 AI，AI 完成接口调用、分页拉取（167条）、语义分类（8大类）。

*关键：提供 Cookie/Token，AI 来写接口调用逻辑。不用自己查接口文档，把浏览器请求截图发过去也行。*

---

**阶段 4：手机端响应式 + 细节打磨**

- 「做成移动端也能看」→ AI 加了全套 `@media(max-width:768px)` 响应式
- 「加个 favicon」→ AI 生成了渐变蓝紫 SVG inline favicon
- 「顶部标题别随版本切换」→ AI 精准改一行 JS
- 「功能名旁边加跳转 Cupid 反馈的按钮」→ AI 加了「🔍 查看相关反馈」

---

**阶段 5：接入产品知识库 AI 助手**

基于 1608 篇 Confluence PRD 页面，构建 BM25 检索 + Gemini 回答的 RAG 问答系统，内嵌到 Dashboard 右下角。这部分是独立的 Python 后端服务（Flask，端口 8890）。

*关键：告诉 AI 想要什么效果（能回答 PRD 相关问题），AI 设计方案、写代码、部署服务。*

## 四、踩过的真实坑

### 坑 1：JS 跨 script 块变量访问报错

**现象**：某个 tab 点进去内容空白，控制台报 `isBizFeat is not defined` / `TK is not defined`  
**原因**：页面拆成了三个 `<script>` 块，第二个块里的 `const R=[...]` 、`const TK=[...]`（数据变量）无法被第三个块访问。`const` 声明有块级作用域，不会自动挂到 `window`。  
**修复**：在第二个 script 块末尾加 `window.R=R; window.TK=TK;` 等，暴露到全局。  
**教训**：多次 patch HTML 容易导致函数定义和调用不在同一个 script 块，排查时要看报错的行号属于哪个块。

---

### 坑 2：scp 大文件被 SIGKILL

**现象**：向服务器传文件，超过 ~100KB 就被系统 SIGKILL 掉  
**原因**：OpenClaw 沙箱有出网流量限制  
**修复**：`gzip -c xxx.html > /tmp/rdb.html.gz`，压缩到 44KB 再传，远端 gunzip  
**教训**：大文件不能直接从沙箱传，要在服务器本地操作，或先压缩。

---

### 坑 3：石墨文档解析有大量噪音

**现象**：石墨文档里产品功能条目抓出来很多垃圾，「展示建议」「宣传文案」「售后」等和功能无关的内容全部混进来  
**修复**：写 HARD_NOISE 正则过滤列表，迭代了 5 个版本解析器  
**教训**：石墨文档结构不规范，需要人工审查一遍，告诉 AI 哪些是噪音，哪些是真正的功能条目。

---

### 坑 4：工单 desc 字段含控制字符破坏 JSON

**现象**：加入工单数据后，发版记录 tab 也变空白了——两个 tab 互相影响  
**原因**：工单描述字段包含非法控制字符（`\x00-\x1f`），写入 HTML 的 JSON 解析失败，整个数据对象崩掉  
**修复**：`re.sub(r'[\x00-\x08\x0b\x0c\x0e-\x1f\x7f]', '', s)` 深度清理字符串  
**教训**：外部系统数据不可信，写入 HTML 前必须做二次验证（`json.loads(json.dumps(...))`）。

---

### 坑 5：手机端 overflow:hidden 导致内容不可见

**现象**：PC 端完全正常，手机端页面内容全空白  
**原因**：`body{overflow:hidden}` + 各 view 的 `overflow:hidden` 在手机端切换了显示模式后，内容被裁切  
**修复**：手机端改为 `body{height:auto;overflow:auto}`，各 view 的 overflow 改为 visible  
**教训**：PC 和手机布局逻辑不一样，加响应式后要真机或 DevTools 手机模拟器验证一遍。

---

### 坑 6：Nginx 代理路径双重 /api 问题

**现象**：API 返回 404  
**原因**：页面请求 `${API_BASE}/api/status`，nginx `location /kb-api/` 代理到 `8888/api/`，实际路径变成 `8888/api/api/status`，多了一层 `/api`  
**修复**：页面请求去掉 `/api` 前缀，改成 `${API_BASE}/status`  
**教训**：nginx proxy_pass 末尾有 `/` 时会替换 location 前缀，配置前想清楚最终路径拼接结果。

## 五、听懂这个分享需要知道什么

### 必要背景

| 概念 | 一句话解释 |
|------|-----------|
| 静态页面 | 只有 HTML/CSS/JS，不需要后端服务器，直接在浏览器运行 |
| API | 从 Confluence/石墨「拿数据」的接口，就像数据出水口 |
| 双数据源 | 页面同时从 Confluence 和石墨各取一部分数据合并展示 |
| RAG | 检索增强生成：先搜相关文档片段，再交给 AI 回答，避免幻觉 |

### 不需要知道

- 具体的 JavaScript / Python 语法
- Confluence API 的参数怎么写
- 服务器 Nginx 如何配置

## 六、这套协作方式适合什么

✅ **适合：**
- 工具型、信息聚合类页面（Dashboard、看板、查询工具）
- 只有自己或小团队使用的内部工具
- 需要快速验证想法的原型
- 把分散在多个平台的数据聚合到一起

⚠️ **注意：**
- HTML 反复 patch 容易积累结构性问题，一旦出现奇怪 bug，建议让 AI 重写而非继续 patch
- 外部数据源（工单、石墨等）数据质量不稳定，接入时要预留「数据清洗」的时间
- 遇到 bug 不用自己排查，把报错信息（控制台截图或文字）直接发给 AI
