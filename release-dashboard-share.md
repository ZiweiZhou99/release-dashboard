> 作者：周子威

---

## 一、背景：为什么要做这个

学练机迭代节奏快，版本多（目前已有 22 个版本、159 个技术功能项），而版本信息分散在两个地方：

- **Confluence**：技术功能清单、RD/QA 负责人、测试状态
- **石墨文档**：产品功能说明

每次想看「这个版本上了什么」，要同时打开两个平台，费时费力，也难以全局把握进度。

**目标**：做一个聚合 Dashboard，把两个数据源整合成一个可视化页面，随时查、随时刷新。

---

## 二、项目是什么

**学练机发版管理平台**，一个纯前端静态页面，核心功能：

| 功能模块 | 说明 |
|---------|------|
| 版本列表 | 左侧展示所有历史版本，含时间和功能数 |
| 版本详情 | 技术功能数、完成率、产品功能数、RD/QA 负责人 |
| 技术发版清单 | 功能 → 负责人 → QA 状态 → 备注 → Jira，表格展示 |
| 产品功能 tab | 对应石墨文档的产品说明内容 |
| 全局搜索 | 跨版本搜索功能名/人员 |
| 产品知识库助手 | 内嵌 AI，基于 PRD 文档实时回答问题 |
| 快捷入口 | Confluence / 石墨 / Cupid / 用户工单 / 反馈 直链 |

**数据源**：Confluence（技术侧） + 石墨文档（产品侧），双数据源实时拉取。

## 三、与 AI 协作的过程

### 协作模式说明

这个项目**全程没有手写代码**，通过和 AI（OpenClaw）对话完成。协作方式如下：

```
我描述需求 → AI 生成代码 → 我在浏览器看效果 → 发现问题 → 告诉 AI → AI 修复/迭代
```

这是一种「产品驱动开发」的方式：你不需要懂代码实现，只需要清楚地描述你**想要什么**和**哪里不对**。

---

### 阶段 1：需求定义

**我做的事：** 用自然语言描述想看的信息结构

示例对话：
> 「我想做一个发版 Dashboard，左边是版本列表，右边是版本详情。详情里要有这个版本的技术功能总数、完成数、进行中数量，还有负责的 RD 和 QA 是谁。」

**AI 做的事：** 理解需求，提炼信息结构，输出 HTML/CSS/JS 代码

**关键点：** 不需要写需求文档，直接对话就行。AI 会主动追问不清楚的地方。

---

### 阶段 2：数据接入

**我做的事：** 提供 Confluence 和石墨文档的数据格式/结构

示例对话：
> 「数据来自 Confluence 这个页面，格式是表格，列是：功能、RD、QA、备注、Jira」  
> 「同时还需要接入石墨文档，这里有产品功能说明」

**AI 做的事：** 写数据拉取逻辑、解析字段、处理跨域问题

**关键点：** 数据接入这类有规律的重复工作，AI 效率极高。遇到跨域/鉴权问题，直接把报错截图/文字告诉 AI，它会给出解法。

---

### 阶段 3：功能迭代

每次发现「体验不好」或「想加功能」，直接描述：

- 「左边版本列表太密了，加上日期和功能数」
- 「想加一个搜索框，能跨版本搜功能名」
- 「加一个筛选器，按 QA 人员过滤」
- 「把 Confluence 和石墨的链接都放顶部」
- 「加一个悬浮 AI 助手，能回答 PRD 相关问题」

**关键点：** 每次改动都很小，反馈快。不用攒一堆需求再改，随时想到随时加。

---

### 阶段 4：细节打磨

- UI 风格：「做成 dark/light 双主题」「卡片风格参考 Linear」
- 交互细节：「点击 QA 状态标签可以筛选」「功能名旁边加跳转到 Cupid 反馈的按钮」
- 数据状态：「加一个更新数据按钮，手动刷新」

## 四、听懂这个分享需要知道什么

### 必要背景

| 概念 | 一句话解释 |
|------|-----------|
| 静态页面 | 只有 HTML/CSS/JS，不需要后端服务器，直接在浏览器运行 |
| API | 从 Confluence/石墨「拿数据」的接口，就像一个数据出水口 |
| 双数据源 | 页面同时从 Confluence 和石墨各取一部分数据合并展示 |

### 不需要知道

- 具体的 JavaScript 语法
- 如何部署服务器
- Confluence API 的具体参数

## 五、这套协作方式适合什么场景

✅ **适合：**
- 工具型、信息聚合型页面（Dashboard、看板、查询工具）
- 只有自己/小团队用的内部工具
- 需要快速验证想法的原型

⚠️ **注意：**
- 复杂业务逻辑、高并发系统不适合这样做
- 代码质量和可维护性需要额外关注
- AI 生成的代码需要你自己能看懂大概结构，方便日后维护

## 六、复刻这个项目的步骤

如果你也想做一个类似的工具：

1. **想清楚「我想看什么信息」** — 越具体越好
2. **确认数据源在哪** — Confluence / 石墨 / YONE 工单 / 数据库等
3. **和 AI 对话，先做最简版本** — 能跑起来最重要，再逐步迭代
4. **遇到问题直接截图/报错丢给 AI** — 不需要自己排查
5. **有个地方能访问** — GitHub Pages / 公司服务器 / 个人域名
